### Структура оценки (10 баллов)

Оценка делится на три блока: **Техническая реализация (Git/Структура)** , **ML-Эксперимент (Методология)** и **Презентация/Защита**.

#### Блок 1: Управление проектом и репозиторий (3 балла)

Пример минимального оформления репозитория [здесь](https://github.com/vifirsanova/sample-rag)

| Критерий | Баллы | Чек-лист (Что проверяем?) |
| :--- | :---: | :--- |
| **1.1 Структура репозитория** | 0.5 | [ ] Создан `README.md` с описанием задачи и инструкцией по запуску. Настроен корректный `.gitignore`. Представлен файл с зависимостями (`requirements.txt`). |
| **1.2 Качество кода** | 0.5 | [ ] Код разбит на модули (папки `src/data`, `src/models` и т.д.). Код написан понятно, не содержит ИИ-артефактов, есть комментарии в сложных местах. Нет закомментированных нерабочих кусков кода. |
| **1.3 Командная работа (Git)** | 2.0 | [ ] **Вклад участников:** видна история коммитов всех членов команды. Использовались ветки (задача не делалась одним коммитом в `main`). |

---

#### Блок 2: ML-эксперимент и аналитика (5 баллов)

| Критерий | Баллы | Чек-лист (Что проверяем?) |
| :--- | :---: | :--- |
| **2.1 Экспериментальный холст** | 2.0 | [ ] **Гипотеза:** сформулировано, что предсказываем и чем (Target + Features). **Датасет:** указан источник, описан минимальный EDA (анализ данных). **Подход:** обоснован выбор первой модели (Baseline). |
| **2.2 Пайплайн и инженерия** | 1.5 | [ ] **Препроцессинг:** данные чистятся и кодируются (обработка NaN, нормализация, чистка регулярными выражениями, но только при необходимости и с кратким пояснением, почему выполнялся этот подход). Процесс воспроизводим (скейлеры/векторайзеры сохранены в отдельные переменные, seed зафиксирован). **Разделение:** выборка разделена на Train/Test корректно (учтён дисбаланс если есть). |
| **2.3 Результат и выводы** | 1.5 | [ ] **Метрика:** выбрана адекватная задаче метрика (не просто Accuracy, если у вас дисбаланс).<br>[ ] **Сравнение:** есть Baseline и улучшенная версия (хотя бы 2 итерации экспериментов). **Интерпретация:** написаны выводы ("Почему модель ошибается?", "Что улучшили?"). |

#### Блок 3: Визуализация и защита (2 балла)

| Критерий | Баллы | Чек-лист (Что проверяем?) |
| :--- | :---: | :--- |
| **3.1 Архитектурная схема** | 1.0 | [ ] **Наличие:** в проекте есть схема (Mermaid/скриншот), отражены Data Flow (если уместно) и компоненты системы. **Использование ИИ:** указано, какой промпт использовался для генерации схемы, частей кода (если использовался). |
| **3.2 Защита и тайминг** | 1.0 | [ ] **Презентация:** тайминг в 10-15 минут соблюден, роли участников отражены в презентационном ipynb или на слайдах.* Каждый участник может объяснить свою часть кода и общий результат. |

### Промпты, которые помогут выполнить критерии (шпаргалка) 

**Для схемы архитектуры (Mermaid):**
> "Создай Mermaid-код диаграммы потоков данных (data flow diagram) для проекта по [Ваш кейс, например: предсказанию оттока]. Покажи этапы: загрузка сырых данных, препроцессинг, обучение модели, сохранение артефакта и предсказание. Учти разделение на train/test."

**Для сбора требований и гипотезы (из Experimental Canvas):**
> "Ты — ML-архитектор. Мы решаем задачу [Кейс]. Опиши раздел 'Project Description' по шаблону: 1) Цель проекта; 2) Целевая переменная; 3) Признаки; 4) Предполагаемый тип модели; 5) Основная метрика качества и почему она подходит."

**Для самопроверки Git:**
> "Напиши bash-команду, которая проверит, что я не запушу в Git файлы датасетов, веса моделей и файлы виртуального окружения." *(Ответ должен содержать создание `.gitignore`)*.
